# Terraform Infrastructure

This directory contains all Terraform code for provisioning the Azure
infrastructure that runs the **Marketing Campaign Builder** application.

---

## Directory Layout

```
infra/
├── README.md                    ← you are here
├── modules/                     ← reusable building blocks
│   ├── tags/                    ← standard tag generation
│   └── resource-group/          ← Azure resource group
└── environments/                ← environment-specific root modules
    ├── dev/
    │   ├── main.tf              ← provider + module wiring
    │   ├── variables.tf         ← variable declarations
    │   ├── outputs.tf           ← stack outputs
    │   ├── terraform.tfvars     ← non-sensitive variable values
    │   └── backend.hcl           ← remote-state config (partial backend)
    ├── test/                    ← same structure as dev
    └── prod/                    ← same structure as dev
scripts/
└── bootstrap-state.sh           ← one-time remote-state storage setup
```

---

## Design Decisions

### Layout strategy

Each environment (`dev`, `test`, `prod`) is an independent **root module**
(stack) with its own Terraform state file. Root modules call shared,
parameterised sub-modules under `modules/` to avoid code duplication.

Benefits:

- Blast radius is limited to one environment.
- `plan` and `apply` are scoped to a single environment; no `-target` hacks.
- Environment-specific IAM, network, or SKU differences stay local to the
  environment without polluting shared module logic.

### Variables vs duplicated code

Environments differ **only through variable values** (set in `terraform.tfvars`
and environment-specific secrets). The module code is identical across
environments. This means:

- A behaviour change is made once in the module and surfaced in all envs via
  PR review.
- Dev can intentionally use lower SKUs / fewer instances than prod by changing
  a variable value rather than branching code.

### Remote state

| Attribute | Value |
|-----------|-------|
| Backend | `azurerm` (Azure Blob Storage) |
| Storage account | `mcbtfstate` (set after bootstrap) |
| Container | `tfstate` |
| State key (dev) | `environments/dev/terraform.tfstate` |
| State key (test) | `environments/test/terraform.tfstate` |
| State key (prod) | `environments/prod/terraform.tfstate` |
| Locking | Blob lease (built into `azurerm` backend — no DynamoDB needed) |
| Versioning | Blob soft-delete + versioning enabled |

> All three environments share the **same storage account** to keep the
> bootstrap surface small. Each environment has its own blob (state file),
> so state is fully isolated.

### Authentication

CI/CD uses **OIDC Workload Identity Federation** (`use_oidc = true` in
`backend.hcl`). No long-lived credentials are stored in GitHub secrets — only:

| Secret | Description |
|--------|-------------|
| `AZURE_CLIENT_ID` | App registration / managed identity client ID |
| `AZURE_TENANT_ID` | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Target subscription per environment |

Local operators run `az login` and authenticate via the Azure CLI.

### Naming convention

Resources follow the pattern: `{type-abbreviation}-{project-short}-{env}-{location-short}`

Examples:

| Resource | Name |
|----------|------|
| Resource group | `rg-mcb-dev-eus2` |
| Storage account | `stmcbdeveus2` (lowercase, no hyphens) |
| Service Bus namespace | `sb-mcb-dev-eus2` |
| Key Vault | `kv-mcb-dev-eus2` |
| Container App env | `cae-mcb-dev-eus2` |
| PostgreSQL flexible server | `psql-mcb-dev-eus2` |

> Resource type abbreviations follow the
> [Azure naming convention guide](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ready/azure-best-practices/resource-abbreviations).

### Standard tags

Every resource carries the following tags (generated by `modules/tags`):

| Tag | Example value |
|-----|---------------|
| `environment` | `dev` |
| `project` | `marketing-campaign-builder` |
| `managed_by` | `terraform` |
| `owner` | `<team name>` |
| `component` | `api`, `worker`, `database`, … |

Additional tags can be injected via `additional_tags` in `terraform.tfvars`.

## First-Time Setup (bootstrap)

Before running `terraform init`, create the remote-state storage account:

```bash
./infra/scripts/bootstrap-state.sh \
  --subscription  <subscription-id> \
  --location      eastus2
```

The script creates:

1. Resource group `rg-mcb-tfstate`
2. Storage account `mcbtfstate` (LRS, TLS 1.2 only, no public blob access)
3. Blob container `tfstate`

After the script completes, copy the printed values into each environment's
`backend.hcl` (the file is already pre-filled with default names — update
`storage_account_name` if a random suffix was appended).

## Day-to-Day Operations

### Initialise an environment

```bash
terraform -chdir=infra/environments/dev init \
  -backend-config=backend.hcl \
  -upgrade
```

### Plan changes

```bash
export TF_VAR_subscription_id="<subscription-id>"
export TF_VAR_tenant_id="<tenant-id>"

terraform -chdir=infra/environments/dev plan \
  -var-file=terraform.tfvars \
  -out=tfplan
```

### Apply changes

```bash
terraform -chdir=infra/environments/dev apply tfplan
```

### Destroy an environment

```bash
terraform -chdir=infra/environments/dev destroy \
  -var-file=terraform.tfvars
```

> **Warning:** Never destroy `prod` without explicit change-management
> approval. The GitHub Actions workflow requires a `prod` environment
> reviewer before applying.

## CI/CD Integration

The `.github/workflows/terraform.yml` workflow:

1. **Detects** which environment directories have changed using
   `dorny/paths-filter`.
2. **Plans** every affected environment on every pull request and posts the
   plan as a PR comment.
3. **Applies** every affected environment automatically when a PR is merged
   to `main`.
4. Supports **manual dispatch** (`workflow_dispatch`) to target a specific
   environment and action (`plan` or `apply`).

### Required GitHub Actions secrets (per environment)

Configure these under **Settings → Environments → {dev|test|prod} → Secrets**:

| Secret | Description |
|--------|-------------|
| `AZURE_CLIENT_ID` | OIDC app registration client ID |
| `AZURE_TENANT_ID` | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Subscription for this environment |

For `prod`, add **required reviewers** under
**Settings → Environments → prod → Protection rules** to gate all applies.

## Adding a new module

1. Create `infra/modules/<module-name>/` with `main.tf`, `variables.tf`, and
   `outputs.tf`.
2. Add a `module "<name>"` block to each environment's `main.tf`.
3. Expose environment-specific tuning (SKU, instance count, feature flags) as
   variables in `variables.tf` and set concrete values in `terraform.tfvars`.
4. Open a PR — the CI workflow will plan all three environments automatically.

## Terraform version

Minimum required version: **1.7.0** (set in each root module's
`required_version`).

Provider versions are pinned to `~> 4.0` for `azurerm` (minor-version
floating within the major, which receives non-breaking updates and security
patches automatically).
